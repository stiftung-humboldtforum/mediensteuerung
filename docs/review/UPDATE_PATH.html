Reine Abhängigkeits-Upgrades · minimaler Code-Touch · risikosortiert

📅 Stand: 2026-05-27 🌿 Branch: review/staged-audit-2026-05 🎯 Ziel: CVE/EOL-Risiken eliminieren ohne Code-Refactor
Scope. Nur Dependency-Bumps die ohne oder mit minimalen Code-Änderungen funktionieren. Major-Migrations (Pydantic v1→v2, aiomqtt 1→2, FastAPI 0.95→0.110+, fastapi-users 11→13) sind als BLOCKED markiert und brauchen einen separaten Refactor-Track.

Risiko-Klassifizierung

SAFE Drop-in. Nur Versions-Pin ändern, Container rebuilden. Patch-Level oder konservativer Minor-Bump.
CAUTION Minor-Version-Bump mit Backward-Compatibility-Garantie, aber Re-Test pflicht. Selten unsichtbare Verhaltens-Änderungen.
RISKY Major-Bump ohne bekannte Breaking-Changes für unseren Use-Case — aber Code-Audit + manueller End-to-End-Test nötig.
BLOCKED Major-Bump mit bekannten Breaking-Changes. Code-Refactor nötig. Nicht Teil dieses Update-Pfads. Separat planen.
DROP Package entfernen statt updaten. Dead Dependency.

Test/Rollback-Workflow

Pro Phase-Step

  1. Branch checkout, einzelnen Dockerfile/Compose-Change einpflegen.
  2. docker compose build --no-cache <service> — auf Build-Errors prüfen.
  3. docker compose up -d <service> — auf Container-Health prüfen (achtung: Watchdog ist broken, manuell prüfen).
  4. 5-Test-Suite ausführen (siehe Phase 0).
  5. Bei Fehler: git checkout HEAD~1 -- <file> + Rebuild.
  6. Bei Erfolg: Commit mit Phase-Bezug (z.B. chore(deps): bump requests to 2.32.4 (CVE-2024-35195) [phase 2.1]).

Rollback-Reihenfolge (falls Phase-Block scheitert)

  1. Service stoppen.
  2. git revert der Phase-Commits.
  3. Image-Tag zurückrollen falls Container-Image betroffen.
  4. MongoDB: Snapshot zurückspielen nur wenn Major-Bump betroffen war.
  5. TLS-Material unverändert lassen — Stack ist mTLS-only, Cert-Wechsel wäre eigene Operation.
