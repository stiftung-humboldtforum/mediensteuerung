# Mediensteuerung — Dependency-Audit 2026-06-26

**Frischer Live-Audit gegen den IST-Stand auf `main`** (nicht gegen das veraltete
`UPDATE_PATH.html` vom 2026-05-27). Erzeugt aus 40 Agenten: pro Build-Unit Live-Abgleich
gegen PyPI / npm / Docker-Hub / endoflife.date, CVE-Prüfung gegen den exakten Pin,
Code-Usage-Check im echten Service-Code, plus adversariale Verifikation jeder
RISKY/BLOCKED/major-Empfehlung (17 bestätigt, 8 als "overstated" zurückgestuft).

## TL;DR

Der schwere Refactor-Track des alten Audits (Pydantic v1→v2, FastAPI 0.95→0.137,
fastapi-users 11→15, motor→beanie/pymongo, aiomqtt 1→2, xknx 2→3, pysnmp 4→7,
React 18→19, MUI→9) ist **fertig auf `main`**. Backend-Services sind nahezu vollständig
auf Latest. Was bleibt:

- **1 echter, anwendbarer Laufzeit-CVE**: FastAPI/Starlette host-header auth-bypass → 1-Zeilen-Fix.
- **3 Blocker** mit klaren Wegen: manager-Python-Pin (aiosnmp), tote webOS-TV-Libs, Supply-Chain-Härtung.
- **Frontend long-tail**: tote Pakete droppen + Tooling-Bumps; ein paar BLOCKED (Code-Refactor).
- **1 zeitkritisch**: Debian bookworm Standard-Support endet **2026-07-11** → `tlsgen` auf trixie (drop-in).

Alle anderen gelisteten CVEs sind **bereits im aktuellen Pin gefixt** (requests 2.34.2,
vite ≥8.0.16, mermaid 10.9.6) oder betreffen inerte Dev-Dependencies.

## B. Blocker (mit Lösungswegen)

### B1 — `manager` auf Python 3.11 festgenagelt (aiosnmp)

**Ursache (verifiziert):** `aiosnmp==0.7.2` (letzte Release 2022, unmaintained) hat eine
Rust/PyO3-Native-Extension und liefert Wheels nur bis cp311. Auf 3.12+ findet pip kein
Wheel → Source-Build mit Rust-Toolchain nötig. **Nicht** Python-Inkompatibilität, sondern
fehlendes Prebuilt-Wheel. Genutzt in **genau einer Datei**: `devices/snmp_gude.py`
(SNMP v2c GET/SET gegen Gude-PDUs, winzige API-Fläche: `Snmp(...)`, `async with`,
`client.get/set`, `x.value`).

| Weg | Aufwand | Risiko | Kern |
|---|---|---|---|
| **A — Rust-Toolchain ins Image, aiosnmp behalten** | niedrig (1–2 Dockerfile-Zeilen, 0 Code) | low | Sofort-Unblock auf 3.12/3.13, identisches Runtime-Verhalten, kein SNMP-Re-Test. Aber: bleibt auf toter Lib, größeres Build-Image (Multi-Stage empfohlen). |
| **B — Migration auf pysnmp 7.x** | mittel (~½ Tag, 1 Datei + Gude-Hardware-Test) | medium | Entfernt die einzige native/unmaintained Lib, manager wird pure-Python, **gleiche Lib wie `fac`** (pysnmp 7.1.27). pysnmp-HLAPI ist kein Context-Manager → `__init__`/GET/SET umschreiben, varBind-Extraktion, Error-Modell. |
| C — puresnmp 2.x | mittel | medium | Pure-Python, aber **neue** Lib ohne Repo-Präzedenz. Kein Vorteil ggü. B. |
| D — net-snmp C-Bindings | hoch | high | Reintroduziert native Systemdep + nicht-async. Nicht empfohlen. |

**Empfehlung:** **A jetzt** (Blocker sofort weg, null Risiko), **B danach** geplant
(aiosnmp endgültig retiren, auf eine SNMP-Lib konsolidieren). C/D meiden.

### B2 — Tote webOS-TV-Libs (`ws4py` + `wsaccel` + `PyWebOSTV`)

**Ursache (verifiziert):** `PyWebOSTV==0.8.9` (2021, latest) hard-depended auf `ws4py==0.6.0`
(tot, 2018, "on hiatus") + `wsaccel==0.6.7` (unmaintained, kompiliert nativ). Genutzt in
**einer Datei**: `devices/tv.py` (`LGWebOSTV`), Fläche winzig — nur `wake` (WOL) + `shutdown`
(`power_off`). `ws4py`/`wsaccel` existieren **nur** für PyWebOSTV; nichts anderes importiert sie.
PyWebOSTV bietet kein pluggable Transport → ws4py lässt sich nicht isoliert ersetzen.

| Weg | Aufwand | Risiko | Kern |
|---|---|---|---|
| **A — `aiowebostv` (HA-Team), alle 3 droppen** | mittel (1 Datei `tv.py` + Cred-Persistenz, 1 Dockerfile-Zeile) | low | Aktiv maintained (v0.7.5, Aug 2025, Apache-2.0), native asyncio → entfernt `asyncio.to_thread`-Brücken + `wsaccel.patch_ws4py`-Hack. Eine Dep: `aiohttp>=3.11`. Python 3.11 läuft schon. Deckt power/WOL + future volume/input. |
| B — bscpylgtv / aiopylgtv | mittel | medium | Auch async, droppt ws4py — aber weniger aktiv als das HA-Standard-aiowebostv. Kein Vorteil. |
| C — PyWebOSTV behalten, nur ws4py droppen | — | high | **Nicht machbar** — hard-depend, kein pluggable Transport. |

**Empfehlung:** **A.** `tv.py` umschreiben (connect macht Registrierung intern,
`client.client_key` statt store-dict in `/opt/weboscreds.json`; `on_open/on_close` → connect/
disconnect + `register_state_update_callback`; `shutdown()` → `await client.power_off()`).
Real-TV-Pairing/Reconnect-Test nötig.

### B3 — Supply-Chain / Reproduzierbarkeit (kein Lock, kein Hash)

**Repo-Form (entscheidend):** jeder Service ist eigenes Submodule → eine Dockerfile-/Req-
Änderung = Commit im Sub-Repo **+** Pointer-Bump hier. Kein Monorepo-Platz für *ein*
gemeinsames Lockfile; per-Service-Tooling muss über ~7 Python-Repos repliziert werden.
Aktuell: Top-Level exact-pinned, aber **keine** transitive Pins, **keine** Hashes, **kein**
Lockfile irgendwo. `frontend` hat `yarn.lock` (committed!), aber Dockerfile macht `RUN yarn`
ohne `--frozen-lockfile` → mutiert die Lock still.

Priorisierte Wege:

- **P0 (jetzt, 15 min, 1 Datei): mkcert SHA256-Verify** — die einzige unzweideutige Lücke
  (unverifiziertes Executable im TLS-Bootstrap). SHA selbst verifizieren, dann pinnen.
- **P1 (jetzt, 1 Zeile): `frontend/Dockerfile` → `RUN yarn install --frozen-lockfile`** —
  Lock existiert schon, Build ehrt ihn nur nicht. (yarn-classic-Flag, nicht `--immutable`.)
- **P2 (Projekt): uv + uv.lock + `UV_REQUIRE_HASHES`** pro Python-Service — Max-Confidence,
  Standard 2025/26, schnell. ODER **P3: pip-tools `--generate-hashes` + `pip --require-hashes`**
  — pip-nativ, minimale Abweichung von jetziger Konvention, ~95 % des Nutzens. Prototyp auf
  `api` (inline-pin) + `fac` (schon requirements.txt), dann replizieren.
- **P4 (zuletzt, nur mit Renovate/Dependabot): Base-Image-Digest-Pinning** `FROM …@sha256:`.
  Ohne Update-Bot wird Reproduzierbarkeit zu Stale-CVE-Schuld → erst Bot, dann pinnen.
